Added OAuth session persistence so users authorize through the browser ONCE and reconnect silently on later launches (previously the Social SDK re-prompted every run, since `authorize` discarded the refresh token). `@discordkit/native` now owns the whole session lifecycle: `authorize()` returns the full `TokenSet`; `refreshToken()` exchanges a stored refresh token for a fresh one; `startSession()` / `resumeSession()` / `endSession()` orchestrate browser-auth-or-silent-reconnect, refreshing proactively via the SDK's token-expiration callback. Storage is a pluggable `TokenStore` seam (`load`/`save`/`clear`, set via `ClientConfig.tokenStore`) so the runtime-specific backend is the only thing an adapter supplies — shipping two: `fileStore` (pure-Node AES-256-GCM, machine-derived key; safe in a Node SEA) and `keyringStore` at `@discordkit/native/auth/keyring` (OS credential vault via `@napi-rs/keyring`). Auth failures are typed for UI branching: `authorize`/`refreshToken` throw an `AuthorizeError` with a `reason` of `timeout` (the local Discord client didn't respond — it isn't running), `declined` (the user cancelled the prompt — detected via either the `Aborted` error type or the OAuth2 `access_denied` code), or `failed`. The shared result helper now surfaces the SDK's `Discord_ErrorType` + a `timedOut` flag (`ResultError`) so callers can classify any async-op failure.
