"@discordkit/tauri": minor
---

Added @discordkit/tauri: a Tauri adapter that runs @discordkit/native (Discord Social SDK) in a Node sidecar and exposes it to the webview over a typed, bidirectional kkrpc bridge (createSidecar / createClient). The sidecar host and webview client are composed per-domain (mirroring @discordkit/native + @discordkit/electron) so an app bundles only the native code it wires, and live Lobby/Call entities cross as serializable snapshots (from native's `toJSON()`) driven by id-keyed RPC. Snowflake ids are strings throughout `@discordkit/native` (Discord's wire convention) and message timestamps are numbers, so every payload is JSON-serializable as-is — the adapter does no id/bigint conversion of its own; the webview-facing types are the plain native types. No Rust crate — kkrpc rides tauri-plugin-shell; the package ships a scaffolded capabilities.json permissions snippet. Covers the full native surface: rich presence + auth + status/log (core) and the users, relationships, activity-invites, lobbies, messaging, and voice domains. Also ships @discordkit/tauri/signals — framework-agnostic TC39-signal conveniences (statusSignal, devicesSignal, logSignal, lobbyIdsSignal, lobbySignal, isConnectedSignal) plus asyncSignal for pull-only reads, in a separate subpath so signal-polyfill is only bundled when used; it re-exports native's `subscribe`/`Subscription` from the koffi-free `@discordkit/native/subscribe` + `/subscription` subpaths so the webview bundle never pulls the FFI client. The sidecar connection detects a sidecar that dies during startup (e.g. the Social SDK can't be loaded) and throws a `SidecarStartupError` carrying its stderr — surfacing actionable guidance immediately instead of letting the first RPC hang until timeout. `tauriSidecarConnection` defaults the sidecar name to `binaries/discord-sidecar`, matching Tauri's `externalBin` convention and the shipped capabilities.json. Session lifecycle is native-owned: the core bridge gains `logout` (→ `endSession`), `connect` resolves to `startSession` when a `tokenStore` is configured (silent reconnect from stored tokens, else browser auth), and `createSidecar` auto-`resumeSession`s on boot so the app reconnects with no webview involvement — the integrator just passes a `tokenStore` (e.g. native's `fileStore`, which is SEA-safe). Also ships `@discordkit/tauri/keyring` — OS-vault token storage (Windows Credential Manager / macOS Keychain / Linux Secret Service via `tauri-plugin-keyring`): `keyringRelay(service)` exposes the vault to the sidecar (pass to `createClient`'s new `expose` option), and `tauriKeyringStore()` is a `TokenStore` whose load/save/clear travel reverse-RPC back to that relay (the sidecar can't reach the OS vault directly). Optional peer dep `tauri-plugin-keyring-api` so it's only bundled when used; the consuming app adds the `tauri-plugin-keyring` crate + `keyring:allow-*-password` capabilities.
